# PBL 템플릿 260612 구조화 분석 및 설계 개선 제안

분석일: 2026-06-15  
원본 파일: `src/PBL템플릿예시_260612.xlsx`  
분석 목적: 최신 PBL 템플릿을 구조화해 현재 Mili AI PBL 생성/검증 설계에 반영할 수정사항과 개선안을 도출한다.

## 1. 파일 구조 요약

이 XLSX는 이미지나 드로잉 객체가 아니라 셀 데이터 중심으로 구성되어 있다. 내부 시트는 7개다.

| 시트 | 역할 |
| --- | --- |
| 프로젝트개요 | 전체 PBL 프로젝트의 제목, 시간, 팀, 난이도, 목표, 제약조건, 평가기준, 하위 미션 목록 |
| 미션지_1 | 1단계: 문제의 정의 및 데이터의 정의 (1~3주차) |
| 미션지_2 | 2단계: 구조 설계 (4~6주차) |
| 미션지_3 | 3단계: 제작 및 연동 (7~9주차) |
| 미션지_4 | 4단계: 완성(배포) 및 분석 (10~12주차) |
| 프로젝트평가 | 최종 Gate 평가 기준과 PASS/조건부 PASS/FAIL 판정 |
| 과제난이도 | 1~10레벨 난이도 기준 |

핵심 구조는 단순한 `Unit > Mission > Task`가 아니라, **프로젝트 개요 → 주차별 미션지 → 최종 Gate 평가 → 난이도 기준**으로 이어지는 캡스톤형 PBL 템플릿이다.

## 2. 프로젝트 개요 구조

원본 템플릿의 대표 예시는 `AI 기반 군사시설 경계 침입 탐지 시스템 개발`이다.

| 항목 | 값 |
| --- | --- |
| 프로젝트명 | AI 기반 군사시설 경계 침입 탐지 시스템 개발 |
| 총 소요 시간 | 학습 시간 12주, 주당 3시간, 총 36시간 + 공유 시간 3주 |
| 팀 구성 | 3명 |
| 난이도 | 9레벨(고급) |
| 프로젝트 목표 | CCTV와 센서 데이터를 활용한 비인가 침입 자동 탐지 AI 시스템 개발 |
| 최종산출물 | 데이터 명세서, 문제정의서, AI 모델 등 |
| 제약조건 | 추론시간 1초 이내, 클라우드 API 호출 금지, 이벤트 후 3초 이내 경보, AI 판단 근거 제시 |
| 평가기준 | 기능완성도, 알고리즘 이해, 토론 참여도 |
| 하위미션 | 문제/데이터 정의, 구조 설계, 제작/연동, 완성/배포/분석 |

이 구조는 현재 생성기보다 훨씬 프로젝트 운영 중심이다. 특히 시간, 팀 구성, 난이도, 제약조건, 하위 미션 목록이 프로젝트 최상위에 존재한다는 점이 중요하다.

## 3. 미션지 공통 구조

각 미션지는 거의 동일한 섹션 구조를 가진다.

| 섹션 | 설명 | 데이터화 필요성 |
| --- | --- | --- |
| 단계 제목 | 예: `1단계: 문제의 정의 및 데이터의 정의 (1~3주차)` | 미션 순서, 기간, 주차 범위 추출 필요 |
| 차시 개요 | 해당 단계에서 무엇을 하는지 설명 | 학습자/멘토용 요약 |
| 학습목표 | 3개 내외의 역량 목표 | 평가 기준과 연결 |
| 선행 학습 권장 과목 | 필요한 사전 지식 | 난이도 및 추천 학습 경로와 연결 |
| 활용 기술 스택 | Python, Jupyter, OpenCV, PyTorch 등 | 기술 태그/실습 환경과 연결 |
| PBL 문제 | 학생 제시용 문제 상황과 미션 | 학습 화면의 핵심 콘텐츠 |
| 5단계 실행 가이드 | Step별 수행 과업, 세부 활동, 산출물 | Web IDE/학습 진행 체크리스트와 연결 |
| 제출물 | PDF, Notebook, 코드, API, 보고서, 발표자료 등 | 제출 UI와 평가 Rubric의 기준 |
| 평가 기준 | 완성도 70%, 팀워크 30%, PASS/FAIL 문항 | 자동/수동 평가 양식 |
| AI 지시문 가이드 | 허용되는 AI 활용, 금지되는 활용, 준수 원칙 | AI 사용 윤리/부정행위 방지 |

따라서 생성 스키마는 `Task.description/output/assessmentCriteria` 정도로는 부족하다. 최소한 `executionSteps`, `deliverables`, `rubricItems`, `aiUsageGuide`가 별도 객체로 분리되어야 한다.

## 4. 4개 미션의 단계성

### 4.1 1단계: 문제의 정의 및 데이터의 정의

목적:

- 군사시설 침입 탐지 문제를 AI 관점에서 재정의한다.
- CCTV, 센서 로그, 침입 사건 기록, 환경 데이터 등 필요한 데이터 유형을 정의한다.
- 실제 군사 데이터 접근이 어려운 상황을 전제로 공개/대체 데이터셋을 탐색한다.
- EDA를 통해 데이터 품질, 불균형, 라벨링, 프라이버시 이슈를 파악한다.

대표 산출물:

- 문제 정의서
- 데이터 명세서
- EDA 보고서
- 팀 회의록

### 4.2 2단계: 구조 설계

목적:

- CNN, LSTM, Hybrid, Autoencoder, YOLO 등 후보 모델을 비교한다.
- 데이터 전처리 파이프라인을 설계한다.
- 데이터 수집, 전처리, AI 추론, 경보 발생, 모니터링 대시보드를 포함한 시스템 아키텍처를 설계한다.
- 평가 지표와 검증 전략을 정의한다.

대표 산출물:

- 모델 설계 보고서
- 시스템 아키텍처 문서
- 데이터 전처리 파이프라인 설계서
- 평가 전략 문서
- 프로젝트 계획서

### 4.3 3단계: 제작 및 연동

목적:

- 설계한 전처리 파이프라인과 AI 모델을 실제 코드로 구현한다.
- 학습 로그, 지표, 출력 결과를 남긴다.
- 전처리, AI 추론, 경보 모듈을 End-to-End로 통합한다.
- 디버깅, 버전 관리, 통합 테스트를 수행한다.

대표 산출물:

- 전처리/모델/추론 코드
- 학습 보고서
- 모듈 통합 테스트 보고서
- 시연 영상
- 개인 기여도 보고서

### 4.4 4단계: 완성(배포) 및 분석

목적:

- REST API 형태로 침입 탐지 시스템을 배포한다.
- 정상, 침입, 엣지 케이스를 포함한 최소 10개 이상 테스트 케이스를 설계한다.
- Precision, Recall, F1-score, 추론 속도를 측정한다.
- 실패 케이스를 분석하고 개선안을 제안한다.
- 최종 보고서, 발표 자료, GitHub 저장소, 회고록을 제출한다.

대표 산출물:

- 배포된 API 서버
- 종합 테스트 보고서
- 최종 프로젝트 보고서
- 발표 자료
- GitHub 저장소 최종본
- 프로젝트 회고록

## 5. 평가 구조

### 5.1 미션별 평가

각 미션은 `완성도 평가 70% + 팀워크 평가 30%` 구조다. 평가 문항은 PASS/FAIL로 되어 있고, 각 문항마다 PASS 기준이 구체적으로 제시된다.

예:

- 문제를 AI 문제로 재정의했는가?
- 필요한 입력 데이터와 수집 방안을 정의했는가?
- 데이터 품질 문제를 분석하고 개선 방안을 제안했는가?
- 시스템 구조를 설계했는가?
- 성능 목표와 검증 방법을 제시했는가?
- 팀원별 역할과 협업 기록이 존재하는가?

### 5.2 최종 Gate 평가

`프로젝트평가` 시트는 미션 점수 합산이 아니라, 프로젝트 목표 달성 여부를 검증하는 Gate 평가다.

| Gate | 평가 의도 |
| --- | --- |
| Q1 문제 해결의 적합성 | 만든 것이 실제 문제를 해결하는가 |
| Q2 시스템 통합 완성도 | 부분 기능이 아니라 완성된 시스템인가 |
| Q3 성능 및 제약조건 충족 | 현장 투입 가능한 수준인가 |
| Q4 기술적 타당성 및 설명 가능성 | 왜 그렇게 만들었는지 이해했는가 |
| Q5 검증 및 개선 역량 | 테스트 결과를 바탕으로 개선 방향을 도출했는가 |
| Q6 팀 협업 수행 | 실제 개발 조직처럼 일했는가 |
| Q7 최종전달력 | 만드는 것뿐 아니라 전달까지 했는가 |

최종 판정:

- PASS: 7개 항목 모두 PASS
- 조건부 PASS: 6개 PASS + Q3 포함
- FAIL: 핵심 항목 Q1~Q3 중 1개라도 FAIL

이 구조는 현재 PRD의 평가 설계보다 강하다. 특히 `Gate 평가`, `조건부 PASS`, `핵심 항목 fail-fast` 개념을 반영해야 한다.

템플릿은 1~10레벨 난이도 체계를 갖고 있다.

| 7 | 고급 | 실시간 AI 시스템 | 실시간 영상 탐지 |

## 7. AI 활용 가이드 구조

각 미션지는 학생용 AI 지시문 가이드를 포함한다.

공통 구조:

- 허용되는 AI 활용
  - 아이디어 브레인스토밍
  - 데이터셋 탐색
  - 코드 스니펫/디버깅 보조
  - 문서 구조 제안
  - 평가 지표 설명
- 금지되는 AI 활용
  - 전체 보고서 대필
  - 분석 결과 조작
  - 전체 모듈 코드 대리 작성
  - 테스트 결과 조작
  - 팀원 역할/회고 허위 작성
- 준수 원칙
  - AI 활용 내역 명시
  - 코드 실행 및 검증 필수
  - 이해 후 사용
  - 자신의 언어로 재작성
  - 결과 조작 금지

이 섹션은 현재 생성 모델에 반드시 추가해야 한다. Mili AI는 AI 활용 교육 플랫폼이므로, "AI를 어떻게 쓰면 되는가"와 "어디부터 부정행위인가"를 콘텐츠 안에서 명시해야 한다.

## 8. 현재 PBL 설계와의 차이

현재 레포의 생성 스키마는 대략 아래 수준이다.

```text
PblPlan
  courseName
  curriculumName
  subject
  units[]
    missions[]
      tasks[]
        description
        output
        assessmentCriteria
        requiredTags
```

최신 템플릿 기준으로 보면 다음 항목이 빠져 있다.

| 누락 항목 | 영향 |
| --- | --- |
| 프로젝트 개요 | 시간, 팀 구성, 난이도, 제약조건, 하위미션을 관리할 수 없음 |
| 미션지 구조 | 차시 개요, 학습목표, 선행학습, 기술스택, 문제상황이 분리되지 않음 |
| 5단계 실행 가이드 | 학습자가 실제로 따라갈 수행 절차가 부족함 |
| 산출물 묶음 | 보고서, 코드, API, 발표자료 등 제출물 구조가 약함 |
| PASS/FAIL 루브릭 | 평가 기준이 문장 배열일 뿐 판정 가능한 문항 구조가 아님 |
| 팀워크 평가 | 협업 증빙, 역할 분담, 통합 산출물 평가가 없음 |
| 최종 Gate 평가 | 프로젝트 목표 달성 여부를 최종 검증하지 못함 |
| 난이도 기준 | 레벨별 콘텐츠 범위와 평가 가능성을 통제하지 못함 |
| AI 활용 가이드 | AI 사용 윤리, 허용/금지 기준, 출처 표기가 빠짐 |
| 참고 자료/도구 | 학습자가 사용할 논문, 튜토리얼, 도구가 연결되지 않음 |

## 9. 기존 설계 수정 제안

### 9.1 생성 단위를 `PblProject`로 승격

현재의 `PblPlan`은 과목 설계 초안에 가깝다. 최신 템플릿을 반영하려면 최상위 모델을 `PblProject`로 두고, 그 안에 프로젝트 개요와 미션지를 포함해야 한다.

권장 구조:

```text
PblProject
  id
  title
  subtitle
  duration
  teamSize
  difficultyLevel
  goal
  finalOutputs[]
  constraints[]
  evaluationSummary
  missionSheets[]
  finalGateReview
  difficultyReference
```

### 9.2 `Unit/Mission/Task`를 캡스톤 구조에 맞게 재매핑

기존 구조를 완전히 버릴 필요는 없지만, 고급 PBL 생성에서는 아래처럼 매핑하는 편이 자연스럽다.

| 현재 구조 | 최신 템플릿 기준 권장 매핑 |
| --- | --- |
| Subject | Project Overview |
| Unit | Mission Sheet / Phase |
| Mission | Execution Step |
| Task | Activity / Checklist Item |
| assessmentCriteria | Rubric Item |
| output | Deliverable |

즉, UI에서는 `Unit/Mission/Task` 용어를 계속 쓸 수 있더라도 데이터 모델은 `MissionSheet > ExecutionStep > Activity`를 별도로 지원해야 한다.

### 9.3 생성 모드를 2개로 분리

최신 템플릿은 12주, 팀 3명, 9레벨 고급 프로젝트다. 모든 PBL을 이 구조로 만들면 Mili AI의 군 장병 학습 제약과 맞지 않을 수 있다.

권장 생성 모드:

| 모드 | 목적 | 산출물 |
| --- | --- | --- |
| Micro-PBL | 15~60분 단위 학습, 초급/중급 장병, 단일 역량 훈련 | 짧은 문제, 실습, 산출물, 즉시 피드백 |
| Capstone PBL | 4~12주 단위 팀 프로젝트, 중급/고급, 종합 산출물 | 프로젝트 개요, 4개 미션지, 최종 Gate 평가 |

현재 `PBL템플릿예시_260612.xlsx`는 Capstone PBL 모드의 표준 샘플로 삼는 것이 맞다.

### 9.5 프롬프트 수정 방향

현재 프롬프트는 "Unit은 2~4개, Mission은 1~3개, Task는 2~4개"를 생성하게 한다. Capstone PBL 모드에서는 아래 지시가 필요하다.

- 프로젝트 개요를 먼저 생성한다.
- 난이도 1~10레벨 중 목표 레벨을 명시한다.
- 4개 미션지를 주차 흐름에 맞게 생성한다.
- 각 미션지는 차시 개요, 학습목표, 선행학습, 기술스택, 학생 제시용 문제, 5단계 실행 가이드, 제출물, 평가 기준, AI 활용 가이드를 포함한다.
- 평가 기준은 문장형 설명이 아니라 PASS/FAIL 판정 가능한 질문과 기준으로 만든다.
- 최종 Gate 평가는 미션 점수 합산이 아니라 프로젝트 목표 달성 여부를 묻는 7개 내외 관문 평가로 만든다.
- 군 장병 대상 콘텐츠에서는 `교수/동료` 같은 대학 문맥을 `멘토/동료 장병/지휘관 또는 업무 담당자`로 바꾼다.

### 9.6 UI 수정 방향

현재 UI는 생성 결과를 계층/표로 보여주는 수준이다. 최신 템플릿을 반영하려면 결과 화면을 다음 탭으로 나누는 편이 좋다.

- 프로젝트 개요
- 미션지 1~4
- 평가 기준
- AI 활용 가이드
- 난이도/제약조건
- Export

각 미션지 화면은 아래 블록을 고정 순서로 보여줘야 한다.

1. 차시 개요
2. 학습목표
3. 선행 학습
4. 활용 기술 스택
5. PBL 문제
6. 5단계 실행 가이드
7. 제출물
8. 평가 기준
9. AI 지시문 가이드

## 10. 템플릿 자체의 보완 포인트

### 10.1 고급 난이도에 치우쳐 있음

현재 예시는 9레벨 고급이며, CNN/LSTM/멀티모달/실시간 경보/API 배포까지 포함한다. 샘플로는 훌륭하지만 초급 장병 대상 첫 PBL로는 어렵다.

개선:

- 같은 사례를 레벨 3, 5, 7, 9로 재구성한 버전을 만든다.
- 초급은 `데이터 분석/문제 정의`, 중급은 `단일 모델/대시보드`, 고급은 `실시간 시스템/배포`로 나눈다.

### 10.2 군 학습자 문맥으로 문구 조정 필요

4단계에는 `교수, 동료` 같은 대학 수업 문맥이 남아 있다. Mili AI에서는 `멘토, 동료 장병, 지휘관, 업무 담당자`로 바꾸는 것이 자연스럽다.

### 10.3 클라우드 제약이 더 명확해야 함

프로젝트 제약조건에는 `클라우드 API 호출 금지`가 있는데, 참고 도구에는 Google Colab, AWS, Weights & Biases 등 클라우드 기반 도구가 등장한다. 이것은 반드시 구분해야 한다.

권장 구분:

- 금지: 외부 AI 추론 API에 군 데이터 전송
- 조건부 허용: 공개/합성 데이터 기반 학습 실습 환경
- 별도 검토: 실데이터, 내부망, GPU 리소스, 실험관리 도구

### 10.4 보안/비식별화 필드가 구조화되어야 함

현재 예시는 실제 군사시설처럼 보이는 설정을 쓰지만, 보안 등급과 비식별화 수준은 별도 필드로 분리되어 있지 않다.

추가 필드:

- 보안 등급
- 비식별화 수준
- 사용 가능 데이터
- 사용 금지 데이터
- 공개 데이터 대체안
- 제출 전 보안 체크리스트

### 10.5 평가에 증빙 필드가 필요함

PASS/FAIL 기준은 좋지만, 왜 PASS인지 기록할 증빙 필드가 없다. 리뷰/평가 도구로 만들려면 각 문항에 `requiredEvidence`, `reviewerComment`, `revisionRequest`를 추가해야 한다.

## 11. 우선 적용할 개선사항

1. **Capstone PBL 스키마를 추가한다.**  
   기존 `PblPlan`을 유지하되, 최신 템플릿용 `PblProject`/`PblMissionSheet` 스키마를 별도로 만든다.

2. **생성 모드를 분리한다.**  
   `빠른 Micro-PBL 생성`과 `12주형 Capstone PBL 생성`을 구분한다.

3. **프로젝트 개요와 최종 Gate 평가를 생성한다.**  
   현재 생성 결과는 중간 미션 구조만 있고 최종 관문 평가가 없다.

4. **AI 활용 가이드를 필수 생성한다.**  
   허용/금지/준수 원칙을 콘텐츠마다 생성하고, 학습 화면에서도 노출한다.

5. **난이도 기준을 제품 데이터로 편입한다.**  
   `과제난이도` 시트의 1~10레벨을 생성 옵션과 품질 검증 기준으로 사용한다.

6. **보안/클라우드 제약 체크를 자동화한다.**  
   `클라우드 API 금지`와 `클라우드 실습 도구 사용`처럼 충돌 가능성이 있는 문구를 자동 검출한다.

7. **미션지 Export 포맷을 지원한다.**  
   생성 결과를 JSON뿐 아니라 미션지형 Markdown/TSV/XLSX로 내보내야 한다.

## 12. 결론

`PBL템플릿예시_260612.xlsx`는 기존 레포의 단순 과정 생성보다 훨씬 구체적인 PBL 설계 기준을 제공한다. 특히 프로젝트 개요, 주차별 미션지, 5단계 실행 가이드, PASS/FAIL 평가, 최종 Gate, AI 활용 가이드는 반드시 현재 설계에 흡수해야 한다.

다만 이 템플릿은 고급 캡스톤에 가깝다. Mili AI의 실제 서비스 설계에서는 이 템플릿을 **Capstone PBL 표준**으로 채택하고, 군 장병의 제한된 학습 시간을 고려한 **Micro-PBL 표준**을 별도로 설계하는 것이 가장 안전하다.
