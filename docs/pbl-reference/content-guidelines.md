# Mili AI PBL 콘텐츠 생성 유의사항

이 문서는 Gemini가 Mili AI PBL 콘텐츠를 생성할 때 반드시 지켜야 하는 **기획 기준과 품질 기준**이다.
콘텐츠는 단순 강의 목차가 아니라, 군 장병이 실제 문제를 해결하는 경험을 하도록 설계되어야 한다.

---

## 1. Mili AI 콘텐츠의 목적

Mili AI PBL 콘텐츠는 군 장병이 복무 중 AI와 데이터 활용 역량을 기르고, 실제 부대 문제를 더 잘 이해하고 해결할 수 있도록 돕는 교육 콘텐츠다.

목표는 전문가 AI 개발자 양성이 아니다.
목표는 다음에 가깝다.

- AI와 데이터가 어떤 문제 해결에 쓰이는지 이해한다.
- 군 실무 문제를 데이터/AI 문제로 재정의할 수 있다.
- 간단한 분석 결과를 읽고 판단 기준을 세울 수 있다.
- AI 교관과 대화하며 문제 해결 과정을 점검할 수 있다.
- 동료와 피드백을 주고받으며 산출물을 개선할 수 있다.

---

## 2. 학습자 전제

주요 학습자는 일반 군 장병이다.

전제 조건:

- 비전공자가 많이 포함된다.
- AI, 데이터 분석, 코딩 경험이 부족할 수 있다.
- 학습 가능 시간이 제한적이다.
- 스마트폰 중심 학습 환경이 많다.
- PC를 사용할 수 있는 시간이 제한적이다.
- 개인 노트북이나 고성능 개발 환경을 전제로 하지 않는다.
- 군 내부 데이터와 개인정보를 직접 다루면 안 된다.

따라서 콘텐츠는 짧고 명확한 단계로 구성해야 한다.

---

## 3. 학습 환경 기준

### 모바일 학습

모바일에서 적합한 활동:

- 상황 카드 읽기
- 핵심 문제 선택
- 이해관계자 선택
- 데이터 컬럼 의미 매칭
- 분석 결과 해석
- 짧은 서술형 입력
- 체크리스트 확인
- AI 교관에게 질문 작성
- 동료 피드백 작성
- PASS/FAIL 기준 확인

모바일에서 피해야 할 활동:

- 긴 코드 작성
- 복잡한 개발 환경 설정
- 여러 파일을 동시에 열어야 하는 작업
- 긴 보고서 작성
- 복잡한 터미널 명령 수행
- 대량 데이터 직접 처리

### PC 학습

PC가 필요한 활동:

- CSV 파일 직접 열기
- Python/pandas 코드 실행
- Notebook 또는 IDE 사용
- 데이터 시각화 생성
- 파일 업로드
- 대시보드 또는 간단한 웹앱 확인
- 최종 산출물 정리

PC가 필요한 경우 `required_device`와 `device_target`을 `pc`로 표시하고, 실행/검증 활동은 `pc_verification` block_type을 사용한다.
PC 없이도 수행할 수 있는 대체 활동이 있으면 `pc_alternative` 또는 별도 step으로 제공한다.

---

## 4. PBL 설계 원칙

PBL 콘텐츠는 기술 목차가 아니다.

나쁜 예:

- Python 기초 배우기
- pandas 사용법 익히기
- 머신러닝 모델 만들기

좋은 예:

- 급식 과발주와 부족 문제를 데이터로 판단하기
- 드론 비행 경로 로그를 보고 위험 구간을 찾기
- 장비 정비 이력에서 반복 고장 패턴을 발견하기
- 행정 문서 분류 기준을 AI 교관과 함께 점검하기

PBL은 항상 다음 흐름을 가진다.

1. 군 실무 문제 상황 제시
2. 학습자가 문제를 자신의 언어로 이해
3. 데이터 또는 AI 활용 가능성 탐색
4. 판단 기준 또는 해결 방법 설계
5. 산출물 제출
6. 동료 또는 AI 교관 피드백
7. 개선 방향 정리

---

## 5. 미션 구성 기준

미션은 2~4개로 구성한다.

- 2개: 초급 / 2~4주 / 데이터 탐색, 간단한 분석, 보고서 작성 중심
- 3개: 초급~중급 / 4~6주 / 데이터 전처리, 분석, 시각화 또는 간단한 예측 포함
- 4개: 중급~고급 / 6~12주 / 설계, 구현, 검증, 발표까지 포함하는 팀 프로젝트

미션 개수는 무조건 4개로 고정하지 않는다.

미션은 다음 역할 중 일부를 단계적으로 담당한다.

- 문제 이해
- 상황 분류
- 데이터 이해
- 기준 설계
- 간단한 분석 또는 실험
- 결과 해석
- 산출물 제출
- 피어리뷰
- 개선안 작성

---

## 6. Step 설계 기준

Step은 실제 화면에 들어갈 수 있는 학습 활동 단위다.

한 step은 한 가지 명확한 활동만 요구한다.

좋은 step:

- 상황 카드 3개를 읽고 가장 가까운 문제 유형을 선택한다.
- 데이터 컬럼과 의미를 매칭한다.
- AI 교관에게 사용할 질문을 작성한다.
- 분석 결과 그래프를 보고 발주 위험 수준을 선택한다.
- 제출물 체크리스트를 확인한다.

나쁜 step:

- 데이터를 분석하고 모델을 만들고 보고서를 작성한다.
- Python을 배워서 전체 시스템을 구현한다.
- AI 모델을 최적화하고 배포한다.

Step은 가능한 한 모바일에서 수행 가능한 활동으로 쪼갠다.
PC가 필요한 작업은 별도 step으로 분리한다.

---

## 7. 학생 노출 문구와 내부 메모 분리

학생에게 보여줄 문구와 내부 검토용 메모를 반드시 분리한다.

학생 노출 문구:

- `learner_text`
- `student_instruction`
- `evaluation_text`

내부 검토용 문구:

- `planner_note`
- `developer_note`
- `planner_review_points`

학생 노출 문구는 쉽고 직접적으로 작성한다.
내부 메모는 기획자와 개발자가 검토할 수 있도록 구체적으로 작성한다.

---

## 8. 예상 답안과 평가 기준

정답이 하나로 고정되지 않는 활동도 많다.
이 경우 `expected_answer_text`에는 “정답”이 아니라 “기대 기준”을 작성한다.

예:

- 문제 상황을 구체적인 부대 업무 장면과 연결했는가
- 데이터 또는 AI가 필요한 이유를 설명했는가
- 성공 기준이 판단 가능한 문장으로 작성되었는가
- 개인정보나 실제 군 내부 데이터를 요구하지 않았는가

선택형/매칭형/순서 배열형처럼 기대값이 있는 경우에는 options에서 `is_expected`와 `expected_order`를 사용한다.

---

## 9. PASS/FAIL 평가 원칙

Mili AI PBL 평가는 점수 경쟁보다 PASS/FAIL 기반 검토에 가깝다.

평가 기준은 다음처럼 작성한다.

- 학생이 이해할 수 있는 문장이어야 한다.
- 평가자가 확인 가능한 기준이어야 한다.
- 산출물 또는 입력값을 보고 판단 가능해야 한다.
- 너무 추상적인 표현을 피한다.

좋은 PASS 기준:

- 문제 배경, 현재 한계, AI/데이터 활용 필요성이 모두 작성되어 있다.
- 주요 데이터 컬럼 5개 이상을 올바른 의미와 매칭했다.
- 제출물에 성공 기준과 제약조건이 포함되어 있다.

나쁜 PASS 기준:

- 잘 작성했다.
- 충분히 고민했다.
- 창의적이다.

---

## 10. AI 교관 활용 기준

AI 교관은 정답을 대신 작성하는 도구가 아니라, 학습자가 생각을 정리하고 검토하는 보조 도구다.

AI 교관 step에서 적합한 활동:

- 문제 정의가 충분한지 질문하기
- 데이터 컬럼 해석을 점검하기
- 분석 방향을 비교하기
- 제출물 개선점을 질문하기
- 피어리뷰 전 자기 점검하기

금지 또는 주의해야 할 활동:

- 전체 보고서를 대신 작성하게 하기
- 전체 코드를 완성본으로 작성하게 하기
- 학습자가 이해하지 못한 결과를 그대로 제출하게 하기
- 실제 군 내부 데이터나 개인정보를 입력하게 하기

AI 교관 관련 step에는 가능한 한 `ai_tutor_question` block_type을 사용한다.

---

## 11. 피어리뷰 기준

피어리뷰는 시험식 채점이 아니라 학습 대화와 개선을 유도하는 장치다.

피어리뷰에 적합한 질문:

- 문제 정의가 실제 군 실무 상황과 연결되어 있는가?
- 제출물만 보고 의사결정 과정을 이해할 수 있는가?
- 성공 기준이 판단 가능한 형태인가?
- 어떤 부분을 보완하면 더 좋아지는가?

피어리뷰 step은 `peer_review_request`를 사용한다.

---

## 12. 데이터와 보안 기준

콘텐츠는 실제 군 내부 데이터, 개인정보, 보안 민감 정보를 요구하지 않는다.

대신 다음을 사용한다.

- 가상 데이터
- 공개 대체 데이터
- 요약 통계표
- 샘플 CSV
- 시나리오 카드
- 비식별 예시 데이터

민감한 데이터가 필요한 주제라면 반드시 `constraints`와 `planner_note`에 대체 데이터 사용 기준을 작성한다.

---

## 13. 기술 스택 사용 기준

기술 스택은 참고 기술 사전에 있는 기술명을 우선 사용한다.

단, 기술 자체가 목적이 되면 안 된다.

나쁜 예:

- pandas를 학습한다.

좋은 예:

- 급식 발주 데이터의 품목별 사용량을 집계하기 위해 pandas를 사용한다.

기술은 항상 문제 해결 행동과 연결해서 작성한다.

---

## 14. 콘텐츠 톤앤매너

학생 노출 문구는 다음 기준을 따른다.

- 친절하지만 과하게 캐주얼하지 않게 쓴다.
- 군 장병이 이해할 수 있는 현실적인 표현을 사용한다.
- 어려운 기술 용어는 쉬운 설명을 함께 제공한다.
- 한 문장이 너무 길어지지 않게 한다.
- “학습한다”보다 “선택한다”, “분류한다”, “작성한다”, “비교한다”, “제출한다”처럼 행동 중심으로 쓴다.

내부 메모는 다음 기준을 따른다.

- 왜 이 step이 필요한지 설명한다.
- 개발자가 어떤 UI 컴포넌트로 구현할지 판단할 수 있게 한다.
- 저장해야 할 데이터가 무엇인지 암시한다.
- 검토가 필요한 지점을 명확히 쓴다.

---

## 15. 금지 사항

다음과 같은 콘텐츠를 생성하지 않는다.

- 단순 VOD 목차
- 기술 키워드만 나열한 목차
- 학생이 모바일에서 수행하기 어려운 긴 코드 입력 중심 콘텐츠
- 실제 군 내부 데이터 입력을 요구하는 콘텐츠
- 개인정보 또는 민감정보 입력을 요구하는 콘텐츠
- 평가 기준이 추상적인 콘텐츠
- 정답이 없는 활동인데 정답을 하나로 단정하는 콘텐츠
- PC가 필요한데 required_device 또는 device_target을 mobile로 표시하는 콘텐츠
- 학생 노출 문구와 내부 메모가 섞인 콘텐츠
- 스키마에 없는 필드를 임의로 추가한 콘텐츠
